# 架构说明

[中文](ARCHITECTURE.zh-CN.md) | [English](ARCHITECTURE.md)

这份文档不是写给只想快速部署的人看的。  
如果你已经能跑起来，想进一步理解“为什么推荐这样部署”，再看这份文档会更合适。

## 一句话理解

这个项目的核心思路是：

不要把 Codex 直接暴露到公网，  
而是在你自己的电脑上本地运行，再通过一个尽量收敛的私有网页入口，让手机只做“查看和聊天控制”。

## 为什么不直接把应用暴露出去

因为你控制的是电脑上的本地 Codex，会话权限通常很高。  
如果暴露方式太粗糙，风险会很大，比如：

- 别人撞库后直接登录
- 公网扫描直接碰到你的服务
- 高风险接口暴露过多
- 新设备拿到账号密码后立刻登录成功

所以这个项目的目标不是“最省事”，而是“自己用时尽量安全、尽量可控”。

## 推荐结构

推荐的运行方式如下：

```text
手机浏览器
   ↓
Tailscale 私网 HTTPS
   ↓
本机 nginx
   ↓
本机 claudecodeui（已应用本项目补丁）
   ↓
本地 Codex 会话
```

## 每一层分别负责什么

### 1. 手机浏览器

它只负责：

- 看项目
- 看会话
- 发消息

它不是完整管理端，也不是远程桌面。

### 2. Tailscale 私网

它负责把访问范围先限制在“你自己的设备网络”里。

这样做的好处：

- 减少公网暴露
- 设备接入和手机访问更简单
- 适合个人长期使用

### 3. nginx

它是网页入口前面的代理层，主要负责：

- 统一入口
- 转发到本地应用
- 附加安全头
- 登录限流
- 让 Node 应用继续只待在 localhost

### 4. claudecodeui + 本项目补丁

这一层负责：

- 提供网页界面
- 提供项目 / 会话 / 消息接口
- 支持手机端交互
- 增加 hardened mode 下的收敛行为
- 增加设备白名单与首次审批能力

### 5. 本地 Codex 会话

真正执行工作的是这里。  
手机不是直接运行 Codex，而是在控制电脑上本地已经存在的 Codex 会话和项目。

## 为什么一定要保留“首次设备审批”

这是这个项目最值得保留的安全边界之一。

如果没有这个机制，那么：

- 只要有人拿到账号密码，就可能直接从陌生设备登录

有了它以后：

- 新设备第一次登录时必须在电脑端批准
- 电脑主人可以看设备名、平台、UA、IP
- 只有批准后的设备才进入白名单

这对“只有自己用”的远程控制面板非常重要。

## 为什么既有 Cookie，又有 Bearer / WebSocket fallback

因为不同手机环境不一样。

### 浏览器环境

浏览器通常最适合走：

- 同源 Cookie 会话

### WebView / 封装 App 环境

一些壳在这些方面兼容不稳定：

- Cookie
- WebSocket
- Header
- 本地存储

所以这里保留了兼容层：

- HTTP 请求可走设备绑定 Bearer fallback
- WebSocket 握手可走 token fallback

但重点是：

- 这些 fallback 是为了兼容，不是为了放宽安全边界
- 新设备仍然要经过白名单审批

## 为什么默认保持 hardened mode

因为目标是手机远程控制，不是远程开放所有能力。

默认收敛后，更适合公开发布和个人私用：

- 攻击面更小
- 不容易误开放高风险功能
- 也更容易给别人讲清楚“这个项目到底能做什么”

## 适合的使用方式

最推荐的使用方式是：

- 只有你一个人在用
- 你的电脑是主执行端
- 你的手机是远程查看和发消息端
- 通过私网访问
- 用桌面工具批准新设备

## 不推荐的使用方式

以下场景不建议直接套这个默认架构：

- 多用户共享
- 公网直接裸奔
- 把它当通用远程执行平台
- 在未重新审计前开放更多高风险接口

## 你真正需要记住的重点

如果你不想看太多技术细节，只需要记住这 4 条：

1. 应用留在本机，不直接公网暴露  
2. 手机通过私网入口访问  
3. 新设备第一次登录必须电脑批准  
4. 手机主要负责“查看和聊天控制”，不要默认开放更多高风险能力
